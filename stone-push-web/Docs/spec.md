## 2. ゲームルール

### 2.1 盤面構成

- 4×4のボードが2×2で合計4枚並ぶ
- 左上と右下が **DARK**、右上と左下が **LIGHT**（対角が同色）
- 上段2枚が白のホームボード、下段2枚が黒のホームボード
- 初期配置は全4ボード共通：黒は一番手前の行（row3）全列に4個、白は一番奥の行（row0）全列に4個
- 4枚のボードの中央交点（黒ホームと白ホームの境目）を「ボーダー」と呼ぶ

### 2.2 ターンの流れ

1ターン＝「リード」と「フォロー」を必ず両方行う。先手は黒。

**リード（パッシブ移動）**
- 自分のホームボード（2枚のうちどちらか）で、自分の石を1つ選ぶ
- 縦・横・斜め（8方向）に1〜2マス動かす
- 経路上（1〜2マス目とも）に石があれば動かせない（押せない・飛び越せない）
- 自分の石をボード外に出すことはできない

**フォロー（アグレッシブ移動）**
- リードで使ったボードと**逆色**のボード（自分・相手どちらのホームでもよい）で行う
- 動かす石を選ぶと、移動方向・歩数はリードと**同じ**に固定されるため、移動先は自動的に1マスに決まる（確認のうえタップで確定）
- 相手の石は1個までなら押し出せる（押さなくてもよい）
- 相手の石を2個以上連続で押すことはできない
- 自分の石を途中や目的地に押す・飛び越すことはできない
- 押し出された相手の石は、攻撃側の最終着地点からさらに1マス先へ移動する。そのマスが盤外ならその石は消滅（復活しない）。盤内かつ石があれば「2個同時押し」として不可

### 2.3 デッドエンドルール

リードした結果、対応するフォローが1手も無い場合、そのリード自体を選ぶことはできない（画面上でも最初から選択肢に出ない）。

### 2.4 勝利条件

4枚のボードのうち、いずれか1枚から相手の石を4個すべて押し出したプレイヤーの勝利。

### 3.2 スタート画面

### 3.3 ゲーム画面

実装済み：
- 4枚のボードを2×2グリッドで表示
- 各ボードの右上相当の位置に残り石数（黒:N 白:N）をリアルタイム表示。いずれかの色が残り1個になったら、その数字を赤字・太字で強調表示（次に押し出されると敗北するため）
- 画面上部にフェーズインジケーター（現在のプレイヤー・フェーズテキスト表示）
- CPU思考中のテキスト表示（盤面グリッドの下に固定表示することで、表示/非表示の切り替え時に盤面セルの位置がずれないようにしている）
- 後攻（白）でプレイ中は、自陣（白ホーム）が画面下側に来るよう盤面の表示順・各ボード内の行の描画順を反転（ゲーム内部の座標・ロジックは反転しない、表示のみの変換）
- 石を動かす／押し出すアニメーション（3.8参照）

フェーズ別ビジュアルフィードバック：

| フェーズ | 表示 |
|---------|---------|
| PASSIVE_SELECT | 動かせる石を枠線でハイライト／動かせない自石は半透明 |
| PASSIVE_CONFIRM | 選択中の石をオレンジ枠でハイライト／移動先候補を枠線表示 |
| AGGRESSIVE_SELECT | 使用不可ボード（パッシブと同色）を暗転／動かせる石を枠線でハイライト |
| AGGRESSIVE_CONFIRM | 使用不可ボードを暗転／選択中の石をオレンジ枠でハイライト／移動先（常に1マス）を枠線表示 |

4枚のボードの中央交点には常時「ボーダー」ラベルを表示（ルール説明画面と同じ用語・同じ位置の視覚的対応）。「ボーダー」＝黒ホームと白ホームの上下の境目のみを太い線で強調し、左右のDARK/LIGHTの区切りは強調しない。

移動先の枠線表示は、押し出し対象（移動先に相手の石がある場合）でも石に隠れず常に見えるようにしている。

未実装：
- PASSIVE_SELECT時の全移動先プレビュー（小ドット）— 選択後にしか出さない
- エラーメッセージのカード表示（デッドエンドはそもそも選択肢に出ないため、選択エラー自体が原理上発生しない設計になっている）

### 3.5 フェーズ制御・操作フロー

```
PASSIVE_SELECT
  └─ 動かせる石をタップ ──────────────────► PASSIVE_CONFIRM
                                               │
                         同じ石を再タップ ◄──┤
                                               └─ 移動先をタップ ──► AGGRESSIVE_SELECT
                                                                        │
                            ハイライト以外をタップ／✕キャンセル ◄─────┤
                            （リードを巻き戻してPASSIVE_SELECTへ）     └─ 動かせる石をタップ ──► AGGRESSIVE_CONFIRM
                                                                                                    │
                                               同じ石を再タップ／✕キャンセル ◄─────────────────────┤
                                               （AGGRESSIVE_SELECTへ戻る）                          └─ 移動先（1マスのみ）をタップ ──► ターン確定
                                                                                                                                        └─ 勝利条件成立 ──► GAME_OVER
```

「フォロー」で動かす石をタップすると、方向・歩数はリードと同じに固定されているため移動先は自動的に1マスへ絞り込まれる（そのマスをタップして確定）。

- CPU手番（vs CPU）：人間側の操作はすべて無効化。CPUは500ms後にパッシブ（リード）→アグレッシブ（フォロー）の順で、同じ速度のスライドアニメーションを再生しながら連続適用

### 3.6 CPU AI

| 難易度 | 状態 | アルゴリズム |
|--------|------|------|
| よわい | ✅ 実装済み | 合法な「リード＋フォロー」の組からランダムに1つ選択 |
| ふつう | ✅ 実装済み | 自分の手のうち、その直後の盤面評価値が最も高いものを選択（1手先読み） |
| つよい | ✅ 実装済み | 自分の各候補手について、相手が最善で応じたと仮定した場合の評価値（2手先読み）を算出し、最も良い手を選択（ミニマックス） |

盤面評価は「石数差×10、ボードごとの全滅寸前ボーナス/ペナルティ±50」のシンプルな評価関数（詳細は`design.md` 5章）。

CPU移動アニメーション（オレンジハイライトの段階表示）は未実装だが、リード・フォローそれぞれのスライドアニメーション（3.8参照）は実装済み。

### 3.7 アンドゥ

未実装。

### 3.8 押し出しアニメーション

実装済み。フォロー（アグレッシブ移動）確定時、動かす石と押し出される石をその場でスライドさせ（`transform: translate`、約400ms）、盤外へ消える石はフェードアウトさせる。リード（パッシブ移動）も同じ400msでスライドする。CPUの手番ではリード→フォローの順に連続してアニメーションが再生される。アニメーション中は操作不可。

### 3.9 設定

言語切替（3.2参照）・CPU難易度選択（3.2参照）以外は未実装。効果音・BGMのON/OFF切り替えや永続化（`localStorage`）は無い。

### 3.10 戦績

`localStorage`によるローカル戦績保存は未実装。オンラインの戦績ランキング（Firebase）は実装済み（3.13参照）。

### 3.11 ルール説明画面

実装済み。1画面の縦スクロールテキスト（盤面構成・ターンの流れ・リード・フォロー・勝利条件・戦績ランキングの6トピックを見出しで区切って表示）。盤面構成のみ、DARK/LIGHTの色分けを再現した簡易図（静的、アニメーション無し）を表示。戦績ランキングのトピックでは「vs CPUで勝利すると結果画面から名前を登録できる（vs人間は登録対象外）」ことを説明する。表示言語はアプリ全体の言語設定に従う。

アクセス経路：
- スタート画面の「ℹ ルール説明」ボタン
- ゲーム画面ツールバーの「ℹ ルール」ボタン（常時表示、ゲーム進行中でも開ける）

どちらから開いても「← 戻る」で元の画面（スタート画面 or 対局中のゲーム画面）にそのままの状態で戻る。

### 3.12 結果画面

実装済み：
- 勝者（黒/白）をオーバーレイ表示
- 人間の勝利時のみ、戦績ランキングへの名前入力フォームを表示（3.13参照）
- 「もう一度」ボタン（同じ設定でリスタート）
- 「メニューに戻る」ボタン

### 3.13 戦績ランキング

実装済み。データ保存先はFirebase（Firestore）。Firebase側の設定内容は`design.md` 10章、戦績データの削除手順は`RECORD_DELETION.md`を参照。

**記録の流れ**：対局終了時、**vs CPUで人間側が勝った場合のみ**、結果画面に名前入力フォームが表示される。vs 人間（パス＆プレイ）およびCPUが勝った場合は記録対象外（フォーム自体が表示されない）。

名前（最大16文字）を入力して送信すると、以下のデータがFirestoreに保存される：

| 項目 | 内容 |
|---|---|
| 名前 | 入力値 |
| カテゴリ | `CPU_EASY` / `CPU_NORMAL` / `CPU_HARD` の3種類（CPU難易度から自動判定。vs人間の記録は無いため`HUMAN`カテゴリは存在しない） |
| 手数 | その対局の成立ターン数（ランキングの主指標、少ない方が上位） |
| 残り石数 | 勝者の4ボード合計の自石残数（参考表示のみ、順位には使わない） |
| 記録日時 | サーバー側タイムスタンプ |

送信は「スキップ」も可能。送信中／完了／失敗の状態を画面に表示する。

**ランキング画面**：スタート画面の「🏆 ランキング」から遷移。カテゴリタブ（CPUよわい／ふつう／つよい）で切替え、各カテゴリの上位50件を手数の少ない順（同数は同順位）で表示する。ページネーションは無し。

## 4. 実装上の設計方針

- **永続化**：ゲーム内設定（言語・CPU難易度）はローカル永続化なし（リロードで既定値に戻る）。戦績データ（vs CPUで人間が勝った対局のみ）はFirebase（Firestore）に永続化（詳細は`design.md` 10章）
- **UIフレームワーク**：無し（素のTypeScript + DOM操作、`innerHTML`による全体再描画方式）
- **状態管理・DI**：状態管理ライブラリやDIコンテナは使わず、`main.ts`のモジュールスコープ変数とモジュール関数の直接importのみで構成
- **多言語対応**：日本語／英語の2言語（詳細は`design.md` 9章）

## 5. 既知の制約・未対応ケース

- 合法なパッシブ移動が1つも無い（完全なデッドロック）状態になった場合、その手番のプレイヤーの反則負け（相手の勝利）として即座に`GAME_OVER`にする（`engine.ts`の`applyAggressiveMove`内で判定）。以前はCPU側の`chooseCpuTurn`が例外を投げて画面が固まる可能性があったが解消済み
- ブラウザでの実機動作確認（クリック操作を含む目視確認）は開発環境上未実施。`npx tsc --noEmit` と `npm run build` の静的チェックのみ通過確認済み
- GitHub ActionsからのGitHub Pagesデプロイの環境保護ルール（`Branch "main" is not allowed to deploy to github-pages`）は解消済み
- CPU「つよい」のミニマックス探索はアルファベータ枝刈りを行っていない全探索のため、盤面の状況によっては候補手数が多くなり、他の難易度よりCPUの応答がわずかに遅くなることがある（実測上は体感できる遅延ではない想定だが、実機での確認は未実施）
- 戦績ランキングはクライアントから直接Firestoreに書き込む方式のため、ブラウザの開発者ツールから偽の戦績を送信することが技術的には可能（ゲーム進行のサーバーサイド検証は行っていない）。`firestore.rules`側で値の範囲・型のバリデーションはしているが、根本対策（Cloud Functions経由での書き込み等）は未実施。詳細は`design.md` 11章を参照
